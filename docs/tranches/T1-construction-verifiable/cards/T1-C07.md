# T1-C07 — Checkpoint et reset

## 1. Résultat attendu

Restaurer exactement le checkpoint initial via `getBase64`/`setBase64`, comparer le hash et réconcilier les listeners.

## 2. Valeur visible pour l'utilisateur ou le jury

La construction devient observable, explicable et vérifiable sans dépendre d'un modèle.

## 3. Prérequis et dépendances

Dépendances : T1-C06. Les preuves de ces cartes doivent être disponibles.

## 4. Périmètre inclus

Restaurer exactement le checkpoint initial via `getBase64`/`setBase64`, comparer le hash et réconcilier les listeners. Sont inclus les types, le cycle de vie, les erreurs et le cleanup nécessaires.

## 5. Contrats, types, entrées et sorties

`Checkpoint{base64,initialHash}` en mémoire; reset exclusif incrémente l'epoch, suspend bridge et invalide les travaux antérieurs.

## 6. Séquence d'implémentation décidée

Capturer après init; annuler; suspendre; restaurer; attendre ready; reconstruire registre/listeners; comparer hash; remettre progrès.

## 7. Cas d'erreur et comportements de repli

Base64 corrompu ou hash divergent: reconstruire A/B/AB depuis fixture canonique et signaler recovery.

## 8. Critères d'acceptation Given/When/Then

Given objets/listeners ajoutés, When Reset, Then le hash initial revient et chaque listener existe une fois.

## 9. Tests et commandes de vérification

À exécuter : tests reset répété/pendant drag et intégration vrai applet round-trip. Puis `pnpm lint`, `pnpm typecheck`, `pnpm test` et, si UI touchée, `pnpm build`. Aucun résultat n'est encore revendiqué.
