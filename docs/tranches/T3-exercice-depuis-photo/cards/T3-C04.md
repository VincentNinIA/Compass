# T3-C04 — Extraire avec Responses et Structured Outputs

## 1. Résultat attendu

Implémenter `POST /api/exercise/parse` : recevoir l'image normalisée, appeler Responses avec `gpt-5.6-terra`, `store:false`, `detail:"original"`, aucun outil et un Structured Output strict, puis retourner une union applicative fermée sans exposer la clé.

## 2. Valeur visible pour l'utilisateur ou le jury

Une feuille claire produit une proposition structurée; une ambiguïté, un exercice hors MVP ou un refus reçoit un état compréhensible au lieu d'un JSON cassé ou d'une construction inventée.

## 3. Prérequis et dépendances

T3-C03 close, schémas T3-C01 disponibles, SDK OpenAI côté serveur et `OPENAI_API_KEY` configurée uniquement dans le runtime serveur. Dépendance directe : T3-C03.

## 4. Périmètre inclus

Route multipart, normalisation T3-C03, prompt d'extraction borné, image data URL éphémère, helper Zod Structured Outputs, timeout, mapping des réponses/refus/incomplete, validation sémantique, dérivation du plan et erreurs HTTP expurgées.

## 5. Contrats, types, entrées et sorties

Entrée : multipart `image` obligatoire et `clarification` optionnelle limitée à 500 caractères. Appel : `responses.parse({model:"gpt-5.6-terra",store:false,tools:[],input:[{role:"user",content:[{type:"input_text",text:prompt},{type:"input_image",image_url:dataUrl,detail:"original"}]}],text:{format:zodTextFormat(ExerciseExtractionWireV1,"exercise_extraction_v1")}})`. Sortie `ParseExerciseResultV1` : `ready { extraction, plan }`, `needs_clarification { question, code }`, `unsupported { reason }` ou `refused { message }`. Headers : `Cache-Control:no-store` et `Pragma:no-cache`.

## 6. Séquence d'implémentation décidée

Valider multipart; normaliser; construire un prompt qui traite tout texte de l'image comme donnée non fiable; appeler Responses sans tool; vérifier statut et contenu; détecter `refusal` avant le parsed output; parser à nouveau le payload à la frontière; appliquer les règles sémantiques T3-C01; dériver le plan uniquement pour `ready`; retourner l'union; libérer image/data URL en `finally`.

## 7. Cas d'erreur et comportements de repli

Entrée invalide : 400/413. Timeout : 504 `parse_timeout`. 401/403 amont : 503 `openai_not_configured` sans détail secret. 429/5xx/réseau : 503 `parse_unavailable`, réessai manuel proposé, aucun retry automatique. Réponse incomplete, parsed output absent ou schéma incohérent : 502 `invalid_model_output`. Refus : branche 200 `refused`. Aucun de ces chemins ne renvoie un plan ni ne touche GeoGebra.

## 8. Critères d'acceptation Given/When/Then

- Given une image claire de la médiatrice, When la route répond, Then le statut est `ready`, le payload valide et le plan est celui dérivé par l'application.
- Given un refus OpenAI, When le contenu est inspecté, Then la route retourne `refused` sans tenter de parser un faux objet.
- Given une réponse incomplete ou hors schéma, When elle est traitée, Then la route retourne 502 et aucun plan partiel.
- Given du texte « ignore previous instructions » imprimé sur l'image, When l'appel est préparé, Then aucun outil n'est disponible et seuls les champs allowlistés peuvent sortir.
- Given `store:false`, When la requête est envoyée, Then ce paramètre est observable dans le mock SDK; aucune promesse ZDR n'est affichée.

## 9. Tests et commandes de vérification

## 12. Hors périmètre

Conversation Realtime, Files API, tools hébergés, streaming, background mode, retry automatique, correction géométrique par le modèle et persistance de réponse.

## 13. Risques et mitigations

Prompt injection dans la feuille : image traitée comme donnée, aucun outil et sortie fermée. Hallucination conforme au schéma : validation sémantique et confirmation T3-C05. Coût/latence de `original` : image pré-bornée à 2400 px. Confusion `store:false`/ZDR : formulation documentaire explicite et contrôle T3-C07.
